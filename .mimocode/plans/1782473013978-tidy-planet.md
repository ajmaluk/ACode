# Fix Dalam Agent Stuck in Thinking State

## Problem Summary

The Dalam IDE agent (Llama 3.3 70B) is stuck in "Thinking" state indefinitely when asked to execute tasks like "list all files in project". The agent never produces tool calls or completes its response.

## Root Causes Identified

### 1. Tool Call Parsing Regex Mismatch (Primary Issue)
**File:** `apps/desktop/src/renderer/lib/dalamAPI.ts` (lines 1478-1779)

The `parseToolCalls()` function uses strict regex patterns that don't match all LLM output formats:
- Regex requires self-closing tags: `<list_dir path="..."/>`
- LLM may output: `<list_dir path="...">` (without self-closing slash)
- LLM may wrap tool calls in markdown code blocks
- LLM may output tool calls with different attribute ordering or whitespace

**Fix:** Make tool call parsing more flexible:
1. Add regex patterns for non-self-closing tags
2. Handle tool calls inside markdown code blocks
3. Add fallback parsing for natural language tool descriptions

### 2. Streaming Not Clearing on Empty Response
**File:** `apps/desktop/src/renderer/lib/dalamAPI.ts` (lines 972-1134)

When the LLM only produces reasoning/thinking content without text output:
- `fullContent` remains empty
- No tool calls are detected
- `message-end` event fires but UI may not properly handle empty response
- `isStreaming` flag stays true indefinitely

**Fix:** 
1. Emit `message-end` even when `fullContent` is empty
2. Add timeout for thinking-only responses
3. Clear streaming state properly on empty responses

### 3. Tool Approval Flow Blocking
**File:** `apps/desktop/src/renderer/lib/dalamAPI.ts` (lines 1781-1853)

The `waitForToolApproval()` function can block indefinitely if:
- Permission dialog doesn't render
- Store subscription fails to detect status changes
- Tool call gets stuck in "pending" state

**Fix:**
1. Add timeout to `waitForToolApproval()`
2. Ensure permission dialog renders for all tool types
3. Add fallback auto-approval for read-only tools

### 4. Missing Tool Type Handling
**File:** `apps/desktop/src/renderer/lib/dalamAPI.ts` (lines 1855-2378)

The `executeTool()` function may not handle all tool types that LLMs output:
- `get_env`, `get_screen_info`, `list_processes`, `kill_process`, `get_disk_space` return "not implemented"
- Some tools may be called with incorrect argument names

**Fix:**
1. Implement missing system tools or provide better error messages
2. Add argument name normalization
3. Add tool call validation before execution

### 5. Model-Specific Tool Calling Behavior
**File:** `apps/desktop/src/renderer/lib/dalamAPI.ts` (lines 244-361)

Different models (Llama 3.3 70B vs GPT-4o vs Claude) have different tool calling capabilities:
- Llama 3.3 70B may not reliably produce XML tool calls
- Some models may use different tool calling formats
- System prompt doesn't adapt to model capabilities

**Fix:**
1. Add model capability detection
2. Adapt system prompt based on model type
3. Add fallback to natural language tool descriptions

## Implementation Plan

   - Handle tool calls inside markdown code blocks
   - Add flexible whitespace handling

   - Handle nested XML structures
   - Extract arguments from various formats

1. **Update `sendPrompt()` loop termination:**
   - Emit `message-end` when `fullContent` is empty
   - Add timeout for thinking-only responses

2. **Update `appendStream()` in store:**
   - Handle empty `streamingContent` in `message-end`

   - Add error recovery for stuck states

### Phase 3: Fix Tool Approval Flow

1. **Update `waitForToolApproval()`:**
   - Add 5-minute timeout
   - Ensure store subscription works correctly
   - Add fallback for stuck approvals

2. **Update permission evaluation:**
   - Auto-allow read-only tools (list_dir, read_file)
